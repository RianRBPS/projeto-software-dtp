1. Contexto e Escopo do Sistema

O trabalho foi desenvolvido seguindo o Tema B — Streaming de Música, composto por dois microserviços isolados: ms-catalogo, responsável pelo acervo de faixas, álbuns e artistas, e ms-reproducao, responsável pela gestão do player, da fila de reprodução e do histórico do usuário.

Toda a gerência de estado é realizada estritamente em memória — sem banco de dados relacional ou NoSQL. A infraestrutura é conteinerizada via Docker, com comunicação entre serviços por HTTP REST.

RestriçãoDecisão
Linguagens permitidasJava, Python ou Node.js (a definir na Fase IV)
PersistênciaSomente em memória (List, Map, estruturas locais)
InfraestruturaDocker + docker-compose.yml
ComunicaçãoHTTP REST (síncrono e assíncrono)
Microserviçosms-catalogo (porta 8080) + ms-reproducao (porta 8081)

2. Fase I — Modelagem Conceitual (Análise de Domínio)

A Fase I aplica a abordagem de Craig Larman para a construção do Modelo de Domínio: somente classes conceituais, atributos em linguagem natural e associações com multiplicidade e verbos de ligação. É terminantemente proibido incluir assinaturas de métodos, modificadores de acesso ou tipos de dados de linguagem de programação.

2.1 Regras de Negócio

RN01 Iterator A fila de reprodução é um agregado dinâmico de faixas vindas de fontes heterogêneas (lista linear de álbum, grafo de recomendações, cache de histórico de busca). O mecanismo de reprodução deve ser agnóstico quanto à estrutura de dados original.

3. Fase II — Modelagem de Arquitetura (Microserviços)

O sistema é decomposto em dois microserviços isolados, cada um com sua própria memória interna e sem compartilhamento de estado. A orquestração é feita via docker-compose.yml, que configura a rede interna dtp-net e expõe as portas HTTP ao cliente externo.

3.1 Decomposição em Microserviços

ContainerPortaResponsabilidadesEstado em memória
ms-catalogo 8080 CRUD de Faixas, Albums, Artistas e GrafoRecomendacao List<Faixa>, List<Album>, List<Artista>, GrafoRecomendacao
ms-reproducao 8081 Gestão do Player, FilaReproducao, EstadoPlayer, HistoricoReproducao e Dispositivos Player, FilaReproducao, EstadoPlayer, List<HistoricoReproducao>

3.2 Protocolo de Comunicação

OrigemDestinoTipoExemplo de rotaPropósito
ms-reproducao ms-catalogo HTTP GET síncrono GET /faixas/{id} Buscar dados de faixas para popular a fila
ms-reproducao ModuloTelemetria HTTP POST assíncrono POST /telemetria/estado Notificar celular e TV do usuário (Observer RN03)
ms-reproducao ModuloHistorico HTTP POST assíncrono POST /historico/registrar Computar dados para a retrospectiva (Observer RN03)
Isolamento de memória: nenhum objeto é compartilhado entre containers. O ms-reproducao mantém apenas o ID da faixa em sua fila; os dados completos (título, BPM etc.) são buscados sob demanda no ms-catalogo via HTTP GET síncrono.

4. Fase III — Design Detalhado (Padrões GoF)

O Diagrama de Classes de Projeto (DCD) é a evolução direta do Modelo de Domínio da Fase I. Nesta fase, todos os atributos recebem tipos de dados, os métodos ganham assinaturas completas, visibilidade é explicitada e os três padrões GoF são aplicados conforme o mapeamento mandatório do DTP.

4.1 Iterator — RN01: Fila de Reprodução Agnóstica

A RN01 exige que o componente que consome a fila de reprodução seja completamente agnóstico quanto à estrutura de dados interna de cada fonte (lista linear de álbum, grafo de recomendações, cache de histórico). O padrão Iterator resolve isso encapsulando a travessia atrás de uma interface uniforme.

Estrutura do padrão

ElementoTipoPapel
FaixaIteratorInterfaceDefine hasNext(), next(), reset() — única interface que o cliente conhece
AlbumIteratorConcretoItera sobre List<Faixa> de um álbum (acesso por índice)
GrafoIteratorConcretoPercorre nós do grafo de recomendações (BFS/DFS), controlando visitados
CacheIteratorConcretoItera sobre o cache de busca linear do histórico
FilaReproducaoContexto/AgregadoAceita fontes distintas e delega a travessia ao iterator configurado
Encapsulamento garantido: o cliente (Player) interage exclusivamente com FaixaIterator — jamais acessa List<Faixa>, List<NoGrafo> ou qualquer outra estrutura interna das fontes.

4.2 Strategy — RN02: Algoritmo de Seleção Intercambiável

A RN02 exige que o comportamento de seleção da próxima música mude dinamicamente, sem reiniciar o container. O padrão Strategy encapsula cada algoritmo em uma classe concreta intercambiável em tempo de execução via Player.definirEstrategia().

Estrutura do padrão

ElementoTipoComportamento
EstrategiaSelecaoInterfaceContrato: selecionarProxima(fila, historico): Faixa
ModoSequencialConcretoSegue estritamente a ordem de inserção; esvazia a fila progressivamente
ModoShuffleConcretoSeleciona aleatoriamente sem repetir no ciclo; mantém conjunto de já tocadas
ModoSmartMixConcretoVarre a fila e seleciona a faixa com menor diferença de BPM em relação à atual
PlayerContextoDelega a seleção à estratégia configurada; expõe definirEstrategia()
Demonstração obrigatória na apresentação: criar uma quarta estratégia (ex: Modo Cronológico por ano de lançamento) e trocá-la via API sem reiniciar o container — evidenciando o princípio Open/Closed.

4.3 Observer — RN03: Sincronização de Estado

A RN03 exige que mudanças no EstadoPlayer sincronizem automaticamente e de forma assíncrona dois módulos independentes. O padrão Observer desacopla o Player (Subject) dos módulos receptores (Observers).

Estrutura do padrão

ElementoTipoPapel
ObservadorEstadoInterfaceContrato: atualizar(estado: EstadoPlayer): void
ModuloTelemetriaObserver ConcretoRecebe estado e envia HTTP POST para cada DispositivoUsuario (celular, TV)
ModuloHistoricoObserver ConcretoRecebe estado e persiste em memória no HistoricoReproducao
PlayerSubjectMantém lista de observers; chama notificarObservadores() em toda mudança de estado

Justificativa: Push vs. Pull

Adotou-se o modelo Push: o Player envia o EstadoPlayer completo para cada observer no momento da notificação. A escolha se justifica por três razões:

  • Redução de round-trips: no modelo Pull, cada observer precisaria fazer uma requisição adicional ao Player para buscar o estado — dobrando o tráfego de rede entre microserviços.
  • Interface mais simples: o contrato atualizar(estado) é autocontido; o observer não precisa manter referência ao subject nem conhecer sua API de leitura.
  • Trade-off aceito: o acoplamento de dados cresce — qualquer campo adicionado a EstadoPlayer é automaticamente propagado a todos os observers, mesmo que eles não o utilizem. Para o escopo deste sistema, onde EstadoPlayer tem apenas três campos (faixaAtual, posicaoMs, statusReproducao), o custo é desprezível.
